# ADR-001: Wybór silnika bazy danych offline (Dexie.js / IndexedDB)

## Status
**ACCEPTED**

## Data
2026-05-18

## Kontekst

FocusFlow 2.0 to aplikacja dla osób z ADHD, która musi eliminować **friction** na każdym etapie. Kluczowe wymagania:

1. **ADHD frictionless UX** - Brak opóźnień sieciowych, brak ekranów ładowania, natychmiastowa reakcja na akcje użytkownika
2. **Offline-first** - Użytkownik musi mieć pełną funkcjonalność bez internetu
3. **Zero logowania** - Eliminacja barier wejścia (paraliż decyzyjny przy rejestracji)
4. **Single-device** - Brak potrzeby synchronizacji między urządzeniami (celowo ograniczony zakres MVP)

### Problemy z klasycznymi rozwiązaniami:

- **Firebase/Supabase** - Wymagają połączenia sieciowego, opóźnienia, autentykacja
- **LocalStorage** - Synchroniczny (blokuje UI), limit 5MB, brak indeksowania
- **SQLite (via WASM)** - Złożona konfiguracja, overkill dla prostych zapytań
- **Raw IndexedDB API** - Verbose, Promise-hell, trudny w utrzymaniu

## Decyzja

Wybór **Dexie.js** jako lekkiego, reaktywnego wrappera na natywne IndexedDB przeglądarki.

### Kluczowe cechy Dexie.js:

```typescript
// Reaktywne zapytania - automatyczny re-render przy zmianie danych
const tasks = useLiveQuery(() => db.tasks.toArray(), []);

// Transakcyjność - atomowe operacje
await db.transaction('rw', db.tasks, async () => {
  await db.tasks.add(newTask);
  await db.tasks.update(oldTaskId, { completed: true });
});

// Łatwa konfiguracja - zero boilerplate
const db = new Dexie('focusflow_v2');
db.version(1).stores({
  tasks: '++id, title, quadrant, subcategory, completed, createdAt'
});
```

## Konsekwencje

### Pozytywne:
- ✅ **Trwałość danych lokalnych** - IndexedDB przetrwa zamknięcie przeglądarki
- ✅ **Transakcyjność** - Atomowe operacje, rollback przy błędach
- ✅ **Reaktywne zapytania** - `useLiveQuery` eliminuje potrzebę manualnego sync UI
- ✅ **Offline-first** - Zero zależności od sieci, działa w tunelu
- ✅ **Zero logowania** - Dane przypisane do urządzenia, nie do konta użytkownika
- ✅ **Wydajność** - Asynchroniczne I/O, nie blokuje wątku UI

### Negatywne:
- ❌ **Single-device limitation** - Brak natywnej synchronizacji (celowo akceptowane)
- ❌ **Browser-dependent** - Różne quota limits (Chrome: ~80MB, Safari: ~1GB)
- ❌ **Brak backupu** - Użytkownik musi samodzielnie eksportować dane (JSON export TBD)

## Alternatywy rozważane

| Rozwiązanie | Powód odrzucenia |
|-------------|------------------|
| Firebase Firestore | Wymaga sieci, autentykacja, opóźnienia |
| LocalStorage | Synchroniczny, limit 5MB, brak struktur |
| SQLite (sql.js) | Overkill, WASM overhead, brak reaktywności |
| RxDB | Zbyt złożony dla MVP, replication niepotrzebne |
