## 2. Kontekst (Context)

### Sytuacja Biznesowa

FocusFlow 2.0 to **mobile-first, offline-first** aplikacja produktywności skierowana do osób z **ADHD i executive dysfunction** (Primary ICP: "The ADHD Brain").

Kluczowe wymagania biznesowe:
- **Privacy-first:** Użytkownicy ADHD często mają wysokie wrażliwość na prywatność (shame cycle związany z "nieproduktywnością")
- **Always-available:** "Object permanence" issues - jeśli app nie działa offline, użytkownik traci dostęp do zadań
- **Zero setup:** ADHD użytkownicy nie mają cierpliwości do konfiguracji kont, logowania, sync
- **Low maintenance:** Solo-dev model - minimalny backend = minimalny devops

### Sytuacja Techniczna

**Wymagania techniczne:**
1. Offline functionality - app musi działać bez internetu
2. Szybkie zapisy/odczyty (<100ms) - ADHD users need immediate feedback
3. PWA-capable - działa jako "natywna" app na mobile
4. Zero backend dependencies - solo-dev friendly
5. Data ownership - user kontroluje swoje dane

**Alternatywy rozważane:**
- LocalStorage / sessionStorage
- IndexedDB (native API)
- IndexedDB + Dexie.js (wrapper)
- SQLite (via WASM/OPFS)
- Backend + REST API (Firebase, Supabase, etc.)

---

## 3. Decyzja (Decision)

**Decydujemy się na użycie IndexedDB jako natywnego magazynu przeglądarki, z Dexie.js jako wrapperem ORM.**

### Konfiguracja:

```typescript
// db/schema.ts
import Dexie from 'dexie';

export class FocusFlowDB extends Dexie {
  tasks!: Dexie.Table<Task, string>;
  notes!: Dexie.Table<Note, string>;
  quizResults!: Dexie.Table<QuizResult, string>;

  constructor() {
    super('FocusFlowDB');
    this.version(1).stores({
      tasks: '++id, quadrant, subcategory, completed, createdAt',
      notes: '++id, taskId, createdAt',
      quizResults: '++id, dominantQuadrant, createdAt'
    });
  }
}

export const db = new FocusFlowDB();
```

**Stack:**
- IndexedDB (native browser API)
- Dexie.js v3.x (Promise-based wrapper)
- No backend / no cloud sync (by design)
- Export/Import JSON (manual backup)

## 4. Uzasadnienie (Rationale)

### Dlaczego IndexedDB + Dexie.js?

#### 1. Offline-First (Must-Have dla ADHD)

ADHD users have **object permanence issues** - jeśli app wymaga internetu, a oni są w miejscu bez zasięgu, "zapominają" o istnieniu zadań. IndexedDB zapewnia pełną funkcjonalność offline.

> "Jeśli nie widzę zadania, przestaje istnieć" - ADHD Brain persona

#### 2. Zero Setup Friction (ADHD-Friendly)

- Brak logowania/rejestracji
- Brak konfiguracji konta
- Brak OAuth flows
- "Open app → start using" w <3 sekundy

ADHD users **nie mają cierpliwości** do setupu. IndexedDB = immediate gratification.

#### 3. Privacy by Design (Shame Cycle Protection)

Dane nigdy nie opuszczają urządzenia:
- Żadnych "produktywności" wysyłanych na serwer
- Żadnego trackingu
- Użytkownik kontroluje 100% danych

Kluczowe dla ADHD users którzy często czują shame związany z "nieproduktywnością" - nie chcą, by ktokolwiek (nawet backend) widział ich "porażki".

#### 4. Solo-Dev Maintainability

- Brak backend do deployowania/monitorowania
- Brak database migrations w chmurze
- Brak auth/security patches
- Brak "unexpected bill" z Firebase/Supabase

**Cost:** $0 infrastructure (vs $20-100/mo dla backend + db)

#### 5. PWA-Ready

IndexedDB działa w:
- Chrome/Edge/Safari/Firefox
- iOS PWA (Safari)
- Android PWA (Chrome)
- Desktop (wszystkie przeglądarki)

Spełnia wymaganie **mobile-first 430px Pro Max** z system_overview.md.

### Dlaczego Dexie.js (wrapper)?

**IndexedDB native API jest:**
- Verbose (boilerplate)
- Callback-based (stare API)
- Brak Promise support
- Brak TypeScript support

**Dexie.js daje:**
- Promise-based API (async/await)
- Full TypeScript support
- Chainable queries
- Automatic indexes
- Transaction support
- Small footprint (~16KB gzipped)

Przykład różnicy:

```javascript
// Native IndexedDB (verbose, callbacks)
const request = indexedDB.open('db');
request.onsuccess = (e) => {
  const db = e.target.result;
  const tx = db.transaction(['tasks'], 'readonly');
  const store = tx.objectStore('tasks');
  store.getAll().onsuccess = (e) => console.log(e.target.result);
};

// Dexie.js (clean, promises)
const tasks = await db.tasks.where('quadrant').equals(1).toArray();
```

### Dlaczego NIE alternatywy?

| Alternatywa | Dlaczego odrzucona |
|-------------|-------------------|
| **LocalStorage** | 5MB limit, synchronous API (blocks UI), no indexing |
| **Native IndexedDB** | Zbyt verbose dla solo-dev, brak TypeScript, callback hell |
| **SQLite (WASM)** | +2MB bundle size, WASM overhead, complex build setup |
| **Firebase/Supabase** | Wymaga auth (setup friction), potential costs, privacy concerns, offline requires sync logic |
| **Custom Backend** | Solo-dev nie może utrzymać 24/7 backend + devops |

## 5. Konsekwencje (Consequences)

### ✅ Plusy (What We Gain)

| Plus | Wpływ na projekt |
|------|-----------------|
| **Zero infrastructure cost** | Solo-dev może skupić się na produkcie, nie devopsie |
| **Immediate startup** | <3s od otwarcia app do użycia (ADHD-friendly) |
| **100% offline** | Works in airplane mode, tunnels, offline home office |
| **Privacy-first** | Zero data leaks, zero tracking, user owns everything |
| **No auth friction** | No "forgot password", no "verify email", no OAuth |
| **Fast queries** | IndexedDB + Dexie indexes = <50ms reads |
| **PWA-ready** | Installable on iOS/Android as "native" app |
| **Simple mental model** | One device = one database. No sync conflicts. |

### ❌ Minusy (What We Lose / Trade-offs)

| Minus | Mitigation |
|-------|------------|
| **No cloud sync** | Użytkownik traci dane przy utracie telefonu | Export/Import JSON (manual backup) |
| **No multi-device** | Działa tylko na jednym urządzeniu | By design - mobile-first 430px Pro Max constraint |
| **Limited storage** | ~60MB limit na iOS Safari | Monitoring usage, alert przy 80% |
| **No sharing/collaboration** | Brak "share tasks with partner" | By design - single-player ADHD focus |
| **No server-side logic** | Brak "AI recommendations" w chmurze | On-device logic only (simpler, private) |
| **No automatic backups** | Użytkownik musi pamiętać o eksporcie | In-app reminders + easy export UX |
| **Browser storage wipes** | "Clear browsing data" = data loss | Education in onboarding + backup reminders |

### ⚠️ Ryzyka do monitorowania

1. **iOS Storage Limit**: Safari na iOS ma agresywne limity (ok. 60MB). 
   - *Mitigation:* Monitor `navigator.storage.estimate()`, alert przy 80% capacity

2. **Data Loss Risk**: Brak automatycznego backupu.
   - *Mitigation:* Export JSON w menu settings + reminder co 7 dni

3. **No Migration Path**: Jeśli w przyszłości potrzeba backendu, migracja będzie trudna.
   - *Mitigation:* Dexie schema wersjonowanie v1, v2, v3... pozwala na future export/import

4. **ADHD Users Forget to Backup**: Paradoks - app dla ADHD, ale wymaga pamiętania o backupie.
   - *Mitigation:* "Set and forget" backup? Brak rozwiązania. To jest trade-off który akceptujemy.

## 8. Podsumowanie

**Decyzja:** IndexedDB + Dexie.js jako storage dla FocusFlow 2.0

**Uzasadnienie:** 
- Offline-first = must-have dla ADHD object permanence issues
- Zero-setup = critical dla ADHD low patience
- Privacy-first = reduces shame cycle associated with productivity tracking
- Solo-dev maintainable = $0 infra, no backend burden

**Trade-off accepted:** 
- No cloud sync → manual JSON backup
- Single-device only → by design for mobile-first 430px Pro Max constraint

**Next Step:** 
- Implementacja w PLAN_001 (Core Data Layer)
- Export/Import JSON feature jako P1 (mitigacja data loss)
